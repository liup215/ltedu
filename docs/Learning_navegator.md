# LTedu「Syllabus Navigator（考纲导航）」最终产品计划文档（全流程学习引擎）

日期：2026-02-17  
面向：产品 / 研发 / 教研 / 运营  
目标：在现有 Syllabus/Chapter/Question/Paper 基础上，构建“基于考纲的全流程学习闭环”。入口无关（Web / App / MCP 等都只是接入层）。

---

## 1. 背景与问题定义

### 1.1 现状亮点
- 系统已具备题库与刷题练习能力，并发布 MCP，使用户可通过大模型进行练习与问答。
- 系统已有考纲（Syllabus）以及章节结构（Chapter），具备做“考纲驱动学习”的内容骨架。

### 1.2 核心痛点
- 对学生“围绕特定考纲”的学习帮助不够：缺少学习规划、过程指导、复习与冲刺的完整流程。
- 当前更偏“用户主动刷题”，缺少“系统主动引导 + 状态跟踪 + 动态调整”的学习引擎。
- MCP 只是入口，不是关键；关键是产品逻辑与学习引擎的全流程闭环。

### 1.3 产品目标
构建��个“考纲导航仪”，帮助学生：
- 选定目标考纲与考试日期 → 得到计划与路线
- 通过前置诊断定位起点 → 进入日常自适应学习循环
- 实时记录学习数据 → 更新掌握度与复习安排
- 自动调整任务与计划 → 保证覆盖率、掌握度、稳定度（记忆保持）
- 最终进入真题/模拟考冲刺 → 诊断→回炉→验证闭环提分

---

## 2. 核心产品哲学（地图 → 导航仪）

### 2.1 三个核心要素
1. **静态地图（Syllabus Knowledge Map）**
   - 以现有 `Syllabus/Chapter` 为骨架
   - 可逐步增强：章节依赖（prerequisites）、核心要点（keypoints）、题目到章节/要点的映射

2. **动态状态（User Knowledge State）**
   - 用户在考纲各部分的掌握度与记忆稳定度（随练习与复习更新）
   - 不是只记录“做没做”，而是记录“会不会、稳不稳”

3. **流式任务（Task Stream）**
   - 不追求一次性生成半年死计划
   - 用“近期可承诺计划 + 远期可预览计划”的方式提供确定性
   - 用户每天看到“今日任务卡片”，同时可看整体进度与未来规划

---

## 3. 完整学习周期（全流程闭环）

> 全周期由四段组成：入驻定位 → 规划 → 日常执行微循环 → 反馈调整与冲刺  
> 每一段都有明确的输入、输出与系统动作。

### 3.1 入驻与目标锚定（Onboarding & Goal Setting）
**目的**：确定终点与约束，定位起点。

#### (A) 考纲订阅 / 学习项目创建（Goal）
用户进入后执行：
- 选择目标考纲（Syllabus）
- 设置：考试日期、目标分数/目标等级、每周可用时长
- 可选：学习模式
  - **同步模式（School Sync）**：以学校进度为最高优先级
  - **自学/冲刺模式（Cram / Self-paced）**：系统规划最优路径

**输出**：一个“学习项目（Goal）”成为后续任务与状态的归属。

#### (B) 前置诊断（Diagnostic / Calibration）
- 生成一份覆盖关键章节的摸底测试（可短可长，MVP 可用少量“锚题”）
- 根据结果初始化用户在考纲各章节的掌握度状态（初始 Knowledge State）

**输出**：初始掌握度热力图、初始复习队列、初始计划生成条件。

---

### 3.2 动态学习规划（Planning）
**目的**：从“目标与起点”生成“近期可执行计划”，并给出“远期方向”。

#### (C) 计划生成的两层结构
- **近计划（硬计划 / 可承诺）**：未来 7 天任务卡片列表（可执行、可调整）
- **远计划（软计划 / 可预览）**：未来 4 周主题与模块目标（不精确到每天）

#### (D) 任务类型（统一任务卡片）
- Learn：学习新知识点（系统可生成微课讲义/学习材料）
- Drill：针对性练习（题目推送）
- Review：复习（间隔重复/SRS）
- Test：阶段测验（章节/模块小测）
- Mock/Past：真题/模拟考（限时）

**输出**：任务卡片流（今日/本周/未来7天），以及“计划版本”用于对比与解释。

---

### 3.3 日常执行与微循环（Execution & Micro-Loop）
**目的**：以“任务卡片”为载体，持续完成“测-学-练-评-改”的闭环。

#### (E) 知识点学习（Learn）
- 系统围绕考纲章节/要点提供学��内容
- 若缺少教材内容，可使用大模型生成“微课讲义”（定义/公式/例题/注意点）
- 学习过程中可进行对话式提问（入口不限）

#### (F) 自适应练习（Drill）
- 根据用户掌握度推送题目
- 练习实时反馈：正确/错误、耗时、错因提示
- 触发“诊断模式（Hint / 引导式纠错）”：优先提示与引导，而非直接给答案

#### (G) 学后检测（Post-check）
- 针对刚学习内容进行短测，判断是否“通关”
- 通关：掌握度上调、安排后续复习点
- 未通关：插入补救任务（更基础讲解 / 降难度练习 / 变式训练）

---

### 3.4 反馈、复习与冲刺（Review & Adapt + Sprint）
**目的**：对抗遗忘、保持稳定、在考前将能力转化为分数。

#### (H) 间隔复习（Spaced Repetition / SRS）
- 系统根据掌握度、时间间隔、错误历史，安排复习任务
- 复习以短任务形式嵌入每日任务流（不是让用户翻错题海）

#### (I) 阶段性检测（Milestone Check）
- 每周/每模块生成小测，校准掌握度
- 防止“平时会做、考试掉链子”的偏差

#### (J) 真题与模拟考冲刺（Sprint）
- 距离考试进入冲刺段：
  - 增加真题/模拟考频率（限时）
  - 将模拟考结果反哺为专项回炉任务（错题与薄弱点）
  - 再用后续真题验证（诊断→回炉→验证闭环）

## 4. 用户可见的进度与未来计划（必须可见）

### 4.1 整体进度（Progress）展示目标
用户能回答：
- 我学到哪了？还剩多少？
- 我是“做过”还是“掌握”？
- 我现在领先/落后吗？风险点在哪？

建议核心指标（UI 可简化文案）：
1. **覆盖率 Coverage**：已触达知识点/章节占比（是否走到）
2. **掌握度 Mastery**：是否真正会做（表现综合）
3. **稳定度 Stability/Retention**：复习后是否仍能做对（是否记得住）
4. **进度预测 Forecast**：按当前节奏预计完成日期与风险提示

### 4.2 未来计划（Plan）展示层级
- **未来 7 天（硬计划）**：每天任务卡片（可调整/可锁定）
- **未来 4 周（软计划）**：每周重点模块与目标（方向感）
- **考前冲刺段（策略计划）**：真题/模拟安排与回炉策略预览

### 4.3 计划更新的可解释性
- 计划要“版本化/快照化”
- 每次调整要告诉用户“为什么改”（例如：某模块错误率高，插入巩固，顺延后续章节）

## 5. 任务与学习规划的更新逻辑（更新架构）

> 推荐 **事件驱动即时更新 + 异步局部重排 + 定时兜底批处理** 的混合架构。

### 5.1 Layer A：即时更新（每次提交后）
触发：用户提交学习数据（做题/完成任务/小测完成）  
执行：
- 写入原始学习日志（Attempt/TaskLog）
- 更新用户掌握度/连续正确/最后练习时间
- 更新复习队列（NextReviewAt）
- 更新当前任务状态（完成/失败/需重试）
- 必要时立即插入少量“补救任务”（局部、即时）

特点：强一致、快反馈。

### 5.2 Layer B：计划局部重算（异步）
触发：关键事件（阶段测验/模拟考/连续落后/目标变更/学校进度变更等）  
执行：
- 异步重排未来 7 天硬计划
- 尊重用户锁定任务
- 生成新计划快照版本，并记录调整原因

特点：弱一致、保证计划合理。

### 5.3 Layer C：后台定时与兜底（cron/批处理）
触发：每日/每周定时或管理员触发  
执行：
- 生成次日任务、补齐任务流
- 周期性重算 4 周软计划
- 处理漏事件、回填数据、算法升级后的校准

特点：最终一致、长期健康。

## 6. MVP 与迭代路线（先跑通闭环，再增强智能）

### 6.1 MVP（第一阶段：跑通全流程）
必须具备：
- 目标创建（选择考纲 + 考试日期）
- 简化诊断（少量题目初始化掌握度）
- 今日任务流（3-5 个任务卡片：Learn + Drill + Review）
- Attempt 记录与掌握度更新
- 基础 SRS（按时间插入复习任务）
- 未来 7 天计划（可预览）

不强依赖：
- 完整 DAG 依赖关系
- 复杂题目难度体系与精细知识点（可后续增强）
- 多入口接入（入口可后置）

### 6.2 第二阶段：增强个性化与可信度
- 考纲增强：章节依赖、核心要点、题目标签（可管理员触发）
- 更细粒度的知识单元（从 Chapter 逐步细化到 Knowledge Point/题型）
- 阶段测验与计划局部重排策略
- 计划版本化与解释系统

### 6.3 第三阶段：冲刺与提分闭环成熟
- 真题/模拟考体系与复盘（错因分类）
- 冲刺策略（高频考点覆盖、时间分配训练、跳题策略）
- 预测与风险模型（更稳定的提分建议）

---

## 7. 非目标与边界说明
- MCP 不是核心，仅为可选接入层之一；产品逻辑必须独立于入口存在。
- 不要求一开始就实现“完美自适应算法”；先用可解释的规则策略跑���闭环，再逐步智能化。
- 计划不是不可变日历：采用“近期可承诺 + 远期可预览 + 可解释调整”的策略。

---

## 8. 最终一句话定义
**Syllabus Navigator 是一个以考纲为中心的学习引擎：用用户状态驱动任务流，用任务流持续更新状态，并在复习与真题冲刺中闭环提分；入口（Web/App/MCP）只是呈现方式。**
